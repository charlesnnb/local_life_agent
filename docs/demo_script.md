# Demo Script (5-7 分钟)

## 推荐展示流程

### 第一阶段：稳定成功展示 (60s) — demo_safe + normal

先用 `APP_MODE=demo_safe` + `demo_scenario=normal` 展示完整成功流程。

> 小明是家住上海徐汇区的用户，有一个 5 岁的孩子。他打开 Agent，输入："今天下午是空的，想和老婆孩子出去玩几个小时，别离家太远，帮我安排一下。"

Agent 的 13 阶段管线：
1. 解析意图 → 识别为家庭场景
2. 确定出发位置 → 用户默认地址（上海市徐汇区）
3. 合并用户画像 → 孩子5岁、3人出行、taxi优先、max 8km
4. 搜索 POI + 餐厅 → 找到若干候选
5. 查询天气 → 晴天 24°C
6. 计算路线 → 各候选距离和时间
7. 评分排序 → 6维度加权评分
8. 可行性检查 → 距离/场景/天气/儿童年龄/门票/空位/排队 12项检查
9. 构建行程 → 出行→活动→晚餐→返程 4段
10. 生成动作计划 → 6项动作（购票/预约/蛋糕/打车×2/发消息）
11. 生成分享消息 → 微信风格短消息
12. 执行工具调用 → 全部成功
13. 生成自然语言解释

展示：
- ✅ Planning Trace 面板：26+ 条 trace 记录，每阶段清晰可见
- ✅ Tool Call Timeline：6 条工具调用全部成功
- ✅ Order Result Card：6 个已完成动作
- ✅ Share Message Box：可一键复制的微信消息
- ✅ Score Breakdown Table：所有候选的 6 维评分分解

### 第三阶段：Fallback 展示 (90s) — 切换 demo_scenario

切换到 `demo_scenario=restaurant_full`：

> 假设最热门的餐厅 17:00 没有空位了。

展示：
- ⚠️ Fallback 记录："17:00 餐厅无位（Demo模拟）"
- ✅ 系统自动尝试 17:30
- ✅ 最终预约成功

切换到 `demo_scenario=rainy_weather`：

> 假设今天是雨天。

展示：
- 🌧 天气强制为中雨
- 📊 POI 天气维度评分：户外活动得分降低 0.2
- ✅ 系统优先选择室内活动

切换到 `demo_scenario=optional_service_fail`：

> 假设蛋糕店今天不营业。

展示：
- ⚠️ 蛋糕下单失败 → 跳过
- ✅ 主行程不受影响
- ✅ 餐厅预约、打车、消息发送仍然成功
- ✅ 分享消息中不提及蛋糕

### 第四阶段：定位展示 (30s) — 用户定位

展示三种定位方式：

1. 点击"📍 使用当前位置" → 浏览器 GPS 定位
   - 成功：显示定位精度
   - 拒绝/超时：优雅 fallback 到默认地址

2. 下拉选择"上海静安区" → 手动输入
   - origin_location.source = "manual"
   - 前端显示："当前出发地：上海市静安区"

3. 不提供任何位置 → 自动使用用户画像默认地址
   - origin_location.source = "profile_default"
   - planning_trace 中说明

## 开场词（30s）

> 我们做的不是本地生活推荐，而是本地生活任务执行 Agent。
>
> 传统推荐系统给你一个列表让你自己挑自己约。我们的 Agent 直接帮用户把事情做完——解析需求、规划路线、检查空位、预约下单、发消息，全部自动完成。
>
> 支持两种演示模式：demo_safe 保证稳定可控，demo_real 接入真实高德地图和 DeepSeek。
> 支持 SSE 流式输出，评委可以实时看到 Agent 每一步在做什么。

## 亮点总结 (30s)

> 1. **不是推荐，是执行** — 完整闭环从需求到订单
> 2. **13 阶段规划管线** — 每一步可解释可追溯
> 3. **真实 API + Mock 双模** — demo_real 展示真实数据，demo_safe 保证不翻车
> 4. **SSE 流式输出** — 实时看到 Agent 思考过程
> 5. **完整 Fallback** — 6 种可控异常场景，无位/无票/雨天/太远都有应对
> 6. **用户定位** — 浏览器 GPS / 手动输入 / 默认地址三选一
> 7. **Provider 抽象层** — 真实 API 可插拔，比赛现场网络不好也不怕
